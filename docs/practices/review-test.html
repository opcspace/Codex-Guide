代码生成结束，不等于任务完成

Codex 的价值来自执行，但最终判断仍属于人。可靠交付至少经过三层：范围审查、行为验证、结果说明。

第一层：看差异

优先检查：

  • 是否出现任务外文件；
  • 是否删掉错误处理或安全检查；
  • 是否引入新依赖、配置或迁移；
  • 是否通过大面积格式化掩盖真实改动；
  • 测试是不是验证行为，而非复制实现。

App 的 review pane 支持逐文件查看、暂存、撤销、提交和推送。也可以输入 /review，选择相对基础分支或未提交变更做只读审查。

第二层：跑最相关的验证

按风险递增：

  1. 直接相关的单元或集成测试；
  2. 类型检查与 lint；
  3. 受影响包或模块测试；
  4. UI 实际操作、截图或浏览器自动化；
  5. 高风险改动的完整套件。

让 Codex报告确切命令和结果，不要只给“测试通过”。如果仓库原本就有失败，必须分开记录。

第三层：形成交付摘要

请用以下结构收尾：
- Outcome：用户可观察到的结果
- Changed：修改文件和关键决策
- Verified：运行命令及结果
- Risks：未覆盖、假设或回滚方式

GitHub 上再做一遍高信号审查

官方 GitHub 集成支持在 PR 中用 @codex review 请求审查，也可配置自动审查和仓库级规则。适合把本地实现与 PR 审查分成两个独立视角。
